# Annexe — Note éditoriale et limites de la version V1.0-draft

## 1. Statut du papier académique

Le papier fourni est une **version académique longue de travail**. Il est structuré, cohérent et exploitable comme base de publication, mais il ne doit pas être considéré comme une version finale prête pour soumission à une revue.

Avant publication externe, il faut encore :

1. vérifier et compléter la bibliographie ;
2. ajouter des références plus spécialisées sur la sociologie des relations internationales ;
3. intégrer 2 ou 3 études de cas courtes ;
4. clarifier le positionnement face à l’École anglaise, au constructivisme et à l’économie politique internationale ;
5. réduire certaines formulations doctrinales trop affirmatives ;
6. ajouter une section méthodologique plus formelle sur la validation empirique.

## 2. Point fort de cette V1

La V1 fixe correctement :

- la thèse centrale ;
- l’unité d’analyse : situation sociale internationale ;
- la notion de personne collective ;
- l’architecture champs/capitaux/conversions ;
- l’importance des dépendances ;
- la causalité configurationnelle ;
- la triangulation probatoire ;
- la compatibilité avec une ontologie JSON.

## 3. Point faible principal

Le point faible principal reste l’absence d’étude de cas développée dans le papier. Pour un texte vraiment convaincant, il faudra appliquer la SIPC sur au moins un cas complet.

Cas recommandé pour la première démonstration :

```text
Détroit de Taïwan
```

Pourquoi : ce cas active presque toutes les briques centrales de la SIPC : souveraineté, reconnaissance, dissuasion, dépendance technologique, hégémonie régionale, audiences internes, alliances, trajectoires et ordre régional.

## 4. Risque conceptuel

Le risque majeur est l’hypertrophie : trop de concepts, trop de couches, trop de vocabulaire.

Règle de correction :

```text
Toute version finale doit forcer la hiérarchie :
une situation,
un mécanisme dominant,
deux à quatre mécanismes secondaires,
des preuves,
des contre-hypothèses,
des signaux d’invalidation.
```

## 5. Recommandation de suite

La suite logique est de produire :

1. le package GitHub Pages réel ;
2. le papier académique resserré avec cas d’application ;
3. les exemples JSON complets ;
4. le prompt d’implémentation LLM ;
5. les tests de validation.
